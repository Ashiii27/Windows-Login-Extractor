Abstract
Windows operating systems generate detailed authentication records across multiple login mechanisms — local accounts, Microsoft accounts, Remote Desktop Protocol sessions, and Active Directory domain authentication. These records are critical forensic artifacts in cybersecurity investigations, incident response workflows, and compliance auditing. Despite their investigative value, extracting and correlating authentication data across different login mechanisms remains a largely manual and fragmented process, with no single native tool handling the complete pipeline. This paper presents WinLogon Forensics — an automated system for extracting, parsing, and analyzing Windows authentication artifacts from Security Event Logs. The system targets key logon-related Event IDs including 4624, 4625, 4634, and 4647, distinguishes between logon types such as interactive, network, and remote interactive sessions, and applies temporal correlation to surface behavioral anomalies including repeated authentication failures, off-hours access, and unusual session patterns. The proposed framework provides security administrators and forensic investigators with a centralized analysis tool that reduces manual effort, improves detection of suspicious access behavior, and generates structured audit reports suitable for incident response and compliance documentation. Experimental results demonstrate the system's capability to process large volumes of authentication log data efficiently while maintaining forensic integrity of source artifacts.
Introduction

Authentication logs generated by Windows systems sit at the center of both security monitoring and digital forensic investigation. Every login attempt — local account, Microsoft account, Remote Desktop session, or Active Directory domain authentication — leaves behind a structured record in the Windows Event Log. These logs store each event that occurs in the operating system, making them among the most reliable sources of evidence when a system is compromised or an unauthorized access event needs to be reconstructed [9]. For forensic analysts, security administrators, and compliance teams, that trail of records is invaluable — it answers the fundamental investigative questions of who accessed a system, when they got in, and how.

The Windows Event Log system has evolved considerably since its introduction. The shift from legacy pre-Vista systems to modern Windows architectures introduced major changes to how authentication telemetry is structured, stored, and queried [2]. Today, authentication events are recorded in the Security Event Log in binary EVTX format, with specific Event IDs carrying distinct forensic meaning. Event ID 4624 records every successful logon, Event ID 4625 captures failed authentication attempts, and Event ID 4634 marks session termination. The logon type field embedded in each record further distinguishes the access method — Type 2 for interactive console sessions, Type 3 for network authentication, and Type 10 for Remote Desktop connections [2]. Together these artifacts form a detailed picture of user activity across a system's lifetime.

Despite their investigative value, working with these artifacts at scale presents a genuine challenge. Event logs are considered one of the primary pieces of evidence in digital forensic analysis, yet extracting and correlating meaningful patterns from them remains a technically demanding task [7]. Windows does not consolidate authentication data in a single location — it distributes records across the Security Event Log, application logs, and registry hives, each following its own schema and retention policy. The native Event Viewer interface was designed for spot-checking individual events, not for investigating access timelines that span multiple systems or weeks of activity.

Automating the recovery, repair, and correlation of multiple event logs makes forensic methods more feasible across a wider range of cases and earlier phases of investigation [8]. However, scripts written for one investigation rarely generalize, coverage is inconsistent, and output typically requires another round of manual interpretation before it is usable. Neither approach scales, and neither is built to surface the attack patterns that only become visible through cross-system, cross-time correlation — credential stuffing campaigns, staged privilege escalation, lateral movement across networked Windows environments.

This paper presents WinLogon Forensics, an automated system for extracting, parsing, and analyzing Windows authentication artifacts across all major login mechanisms. The system targets logon-related Event IDs including 4624, 4625, 4634, and 4647, distinguishes between logon types throughout the pipeline, and applies temporal correlation to flag behavioral anomalies without requiring manual triage. The architecture is designed to facilitate extraction and analysis of operating system artifacts while remaining extensible and reusable across diverse investigative contexts [6]. The output is structured for two primary user groups: security administrators who need anomalies surfaced efficiently, and forensic investigators who need defensible timelines and audit-ready reports.

Literature Review
Role of AI and Automation in Digital Forensics

Event log analysis has been a staple of cybersecurity practice and digital forensics for years. What has shifted recently is the growing role of automation and artificial intelligence in that process. Manually correlating large volumes of log data, spotting anomalies buried across thousands of entries, and reconstructing accurate forensic timelines are tasks that do not scale well with human effort alone — and researchers have increasingly turned to machine-driven approaches to close that gap [1][3].

Deep learning architectures — particularly LSTM networks and Transformer-based models — have shown strong results when applied to system log data, handling complex sequential patterns that simpler rule-based methods tend to miss [2]. Explainable AI techniques are gaining traction specifically because forensic work carries accountability requirements that a black-box output cannot satisfy — findings need to be traceable, understandable, and defensible in contexts where legal reliability matters [4]. Retrieval-Augmented Generation approaches have also been applied to log anomaly detection, combining the contextual reasoning of large language models with structured log retrieval to improve detection accuracy [5].

Windows Event Log Architecture

The Windows Event Log system has gone through meaningful changes. The classic Event Log API shipped with Windows XP and Server 2003 was replaced by the modern EVTX format introduced with Windows Vista — a shift that brought structured XML-based logging and improved querying capabilities. Windows logon forensics requires a thorough understanding of authentication protocols such as Kerberos and NTLM to properly interpret logon process communications [11]. Without this foundational understanding, investigators risk drawing incomplete conclusions from authentication artifacts.

Two Event IDs sit at the center of most authentication analysis work. Event ID 4624 records every successful logon, embedding details about the logon type, authentication package used, and originating network address. Event ID 4625 captures every failed authentication attempt, making it the primary indicator for brute-force activity and credential stuffing campaigns [8].

Authentication Protocol Forensics

Kerberos has been the default Windows authentication protocol since 2000, but there are still scenarios where Windows falls back to NTLM [2]. This fallback behavior has significant forensic implications — the presence of NTLM authentication in an otherwise Kerberos-dominated environment can itself serve as an indicator of suspicious activity. NTLM's challenge-response model allows attackers to capture and relay authentication artifacts, while pass-the-hash and credential theft remain practical lateral movement techniques in breached networks [8].

Log Extraction Techniques

Several approaches exist for extracting Windows Event Logs programmatically. PowerShell's Get-WinEvent cmdlet is the most accessible native option, supporting XML filtering and XPath queries. Python libraries like python-evtx and pywin32 offer a viable alternative, enabling .evtx parsing outside of a Windows host entirely. Timestamped exports and hash verification are routine in forensic work precisely because collected logs may end up as evidence and their integrity will be questioned [6]. Commercial SIEM platforms approach collection differently, but the opacity of their parsing pipelines is a known limitation when investigation methodology itself is subject to scrutiny [3].

Lateral Movement Detection

Lateral movement is a principal tactic in the arsenal of advanced persistent threat groups — it enables an attacker, after establishing a foothold, to maintain ongoing access and penetrate further into a network [12]. JPCERT/CC's landmark study documented the specific Windows Event IDs generated by 44 commonly used lateral movement tools, establishing that network logon events — particularly Type 3 logons accessing administrative shares — serve as reliable indicators when correlated across multiple systems [10]. More recent work leverages the LMD-2023 benchmark dataset, comprising over 1.7 million log samples in EVTX format, to evaluate machine learning approaches for lateral movement detection [14].

Objectives
Primary Objective

This project sets out to design and implement an automated extraction and analysis framework that handles the complete Windows authentication log pipeline — from raw EVTX files through to structured forensic output — without requiring manual intervention between stages. The system targets security administrators who need anomalies flagged without manual triage, forensic investigators who need defensible and reproducible timelines, and compliance teams who need audit-ready reports.

Sub-Objectives
  • Automated Log Extraction: Implement extraction using PowerShell Get-WinEvent for live systems and Python EVTX libraries for offline forensic analysis, with SHA-256 integrity hashing [6].
  • Event Parsing and Normalization: Parse Event IDs 4624, 4625, 4634, 4647, 4778, 4779 and classify logon types into a unified normalized schema [7].
  • Session Correlation: Correlate logon/logoff pairs using LogonID as correlation key to reconstruct complete per-user activity timelines [2].
  • Behavioral Anomaly Detection: Detect brute-force attempts, off-hours access, new source addresses, privilege escalation, and concurrent sessions indicating credential compromise [2][5].
  • Threat Detection and Alerting: Implement detection rules for credential stuffing, lateral movement via Type 3 logons, and NTLM relay indicators with severity classification [3][7].
  • Forensic Timeline Construction: Build chronological timelines with full contextual metadata exportable in forensic tool-compatible formats [1][9].
  • Report Generation: Generate PDF, CSV, JSON, and HTML reports with visualizations for incident response and compliance documentation [3][7].

Methodology
System Architecture Overview

The WinLogon Forensics system is built around a multi-stage pipeline that moves authentication log data from raw extraction through to actionable forensic output without requiring manual intervention between stages. The architecture follows a sequential processing model with six core components: Log Acquisition, Event Parsing and Normalization, Session Correlation, Behavioral Analysis, Threat Detection, and Report Generation.

The pipeline operates in two modes. In live mode, PowerShell interfaces directly with the Windows Event Log service. In forensic mode, Python parses archived EVTX files independently. Both modes produce identical normalized output, ensuring that downstream analysis components behave consistently regardless of the acquisition method used [6][7].

Stage 1 — Log Data Acquisition

The system initiates by identifying target Windows Event Log sources, primarily the Security log channel. For live systems, PowerShell's Get-WinEvent cmdlet is employed with XML filtering to extract Event IDs 4624, 4625, 4634, 4647, 4778, and 4779. The acquisition phase implements integrity verification through SHA-256 cryptographic hashing to ensure evidential validity of extracted logs, consistent with forensic evidence preservation standards [6][8].

Algorithm 1: Windows Event Log Extraction
Input: Target system, Event ID list, Time range Output: Extracted event records + SHA-256 hash 1. Initialize parameters and input configuration 2. Verify administrative privileges If insufficient → terminate and log error 3. Validate Event ID list and time range 4. Identify Security Event Log as primary source 5. Construct XML-based filter query 6. Establish connection with Event Log service 7. For each Event ID in input list: a. Execute Get-WinEvent with XML filter b. Retrieve events in batches c. Apply timestamp filtering d. Parse and store in memory buffer 8. Merge all records into unified dataset 9. Export consolidated events to XML 10. Remove duplicate/corrupted entries 11. Compute SHA-256 hash of export file 12. Store logs and integrity hash securely 13. Generate extraction summary report 14. Return extracted records + integrity hash END
Stage 2 — Event Parsing and Normalization

Raw Event Log XML is parsed to extract structured authentication data. The parser implements field mapping to normalize diverse event structures into a unified schema. Timestamps are converted to ISO 8601 format. Logon type integers are mapped to human-readable labels. Missing fields are handled through null-safe processing [6][7].

Algorithm 2: Event Parsing and Normalization
Input: Raw XML event record Output: Normalized authentication record 1. Parse XML → extract System and EventData 2. Extract timestamp → convert to ISO 8601 3. Retrieve Event ID → map to event type: 4624 → Successful Logon 4625 → Failed Logon 4634 → Logoff 4647 → User-Initiated Logoff 4778 → Session Reconnected 4779 → Session Disconnected 4. Extract username and domain fields 5. Retrieve logon type → map to label: 2 → Interactive 3 → Network 7 → Unlock 10 → RemoteInteractive 6. Extract source IP and hostname 7. Retrieve auth package and LogonID 8. Apply null-safe handling for missing fields 9. Construct and return normalized record END
Stage 3 — Authentication Event Correlation

Authentication events are correlated across time to reconstruct complete user sessions. Event ID 4624 entries are paired with corresponding Event ID 4634 or 4647 records using the LogonID field as a correlation key, enabling precise session duration calculation. For Remote Desktop sessions, Event IDs 4778 and 4779 provide additional temporal markers [2].

Algorithm 3: Session Correlation
Input: List of normalized authentication events Output: Correlated sessions with durations 1. Initialize session dictionary keyed by LogonID 2. Sort all events chronologically 3. For each event in sorted list: If event_type == Logon (4624): → Create session entry with logon timestamp → Store username, source IP, logon type → Add to dictionary by LogonID If event_type == Logoff (4634 or 4647): → Lookup session by LogonID → Calculate duration = logoff - logon time → Mark session as complete If event_type == Reconnect (4778): → Update session with reconnection time If event_type == Disconnect (4779): → Record disconnection marker 4. Flag sessions without logoff as incomplete 5. Return correlated session list END
Stage 4 — Behavioral Analysis

The system performs statistical analysis on authentication patterns to establish per-user behavioral baselines and detect deviations. For each user account, temporal features are extracted including typical login hours, average session duration, preferred authentication methods, and common source systems. Anomaly detection flags deviations such as logins during unusual hours, authentication from new source addresses, sudden increases in failed attempts, privilege escalation events, and simultaneous sessions from different IP addresses [2][5].

Stage 5 — Threat Detection and Alert Generation

Detection rules are applied over the behavioral analysis output to identify specific threat patterns. Three primary threat categories are targeted [3][7]:

Brute-Force Detection flags accounts exceeding a configurable failed attempt threshold within a sliding time window (default: 10 failures within 300 seconds).

Credential Stuffing Detection identifies rapid authentication attempts distributed across multiple accounts from a single source IP.

Lateral Movement Detection analyzes Type 3 Network logon events accessing administrative shares — a pattern consistently associated with lateral movement tooling in enterprise breach investigations [12].

Algorithm 4: Brute-Force Detection
Input: Failed logon events, threshold parameters Output: List of detected brute-force attempts 1. Define: THRESHOLD=10, WINDOW=300 seconds 2. Group failed events by source IP address 3. For each source IP: a. Sort failed attempts by timestamp b. Apply sliding window algorithm: Initialize window at first attempt Count attempts within TIME_WINDOW If count >= FAILURE_THRESHOLD: → Flag as brute-force attempt → Record affected usernames → Calculate attempt rate → Generate Critical severity alert 4. Return list of flagged attempts END
Stage 6 — Timeline and Reporting

The system constructs detailed chronological authentication timelines with full contextual metadata for each event. The reporting module generates output in four formats: PDF for formal forensic documentation, CSV and JSON for SIEM ingestion, and HTML with embedded Plotly visualizations for operational security review [3][7].

from collections import defaultdict from sklearn.ensemble import IsolationForest import pandas as pd def detect_brute_force(failed, threshold=10, window=300): alerts = [] by_ip = defaultdict(list) for ev in failed: by_ip[ev["source_ip"]].append(ev) for ip, attempts in by_ip.items(): s = sorted(attempts, key=lambda x: x["timestamp"]) ws = 0 for i in range(len(s)): ti = datetime.fromisoformat( s[i]["timestamp"]) t0 = datetime.fromisoformat( s[ws]["timestamp"]) while (ti-t0).seconds > window: ws += 1 t0 = datetime.fromisoformat( s[ws]["timestamp"]) if (i - ws + 1) >= threshold: alerts.append({ "type":"Brute Force", "severity":"Critical", "source_ip": ip, "count": i-ws+1 }) return alerts def detect_anomalies(sessions): df = pd.DataFrame(sessions) df["hour"] = pd.to_datetime( df["logon_time"]).dt.hour df["dur"] = pd.to_timedelta( df["duration"].fillna("0:00:00") ).dt.total_seconds()/60 feats = df[["hour","dur"]].fillna(0) model = IsolationForest( contamination=0.05, random_state=42) df["score"] = model.fit_predict(feats) return df[df["score"]==-1].to_dict("records")

Discussion

The experimental results demonstrate that the WinLogon Forensics system successfully addresses the core problem identified in the problem statement. Several observations emerged from testing worth discussing in depth.

Forensic Integrity Preservation — SHA-256 hashing at the point of extraction proved straightforward to implement and adds meaningful evidential value without measurable performance overhead [6].

Logon Type Differentiation — Correctly distinguishing between Interactive, Network, and RemoteInteractive logon types proved critical for meaningful analysis. Events that appear suspicious in isolation require logon type context to be correctly prioritized [8].

Baseline Establishment Overhead — The behavioral anomaly detection component requires sufficient historical session data to establish meaningful per-user baselines. In test environments with limited authentication history, the IsolationForest model produced a higher false positive rate than in environments with richer baseline data.

Conclusion and Future Work
Conclusion

Windows authentication logs represent one of the richest sources of forensic evidence available on any compromised or monitored system. The native tooling available on Windows systems was never designed for investigative use at scale, and the gap between available telemetry and accessible insight has remained a persistent challenge for forensic investigators and security administrators alike.

This paper presented WinLogon Forensics — an automated six-stage pipeline that addresses that gap directly. The system handles extraction from live Windows systems and archived EVTX files, normalization into a unified schema, session correlation using LogonID-based event pairing, behavioral anomaly detection through rule-based and statistical methods, threat detection targeting brute-force attempts, credential stuffing, and lateral movement patterns, and structured report generation across PDF, CSV, JSON, and HTML formats.

Several contributions emerge from this work. First, the system consolidates a pipeline that previously required multiple disconnected tools into a single automated workflow. Second, forensic integrity is enforced at the point of extraction through SHA-256 hashing. Third, the dual-mode architecture supports both live system analysis and offline EVTX forensics. Fourth, the transparent open implementation addresses a key limitation of commercial SIEM platforms whose proprietary pipelines are unsuitable for investigations where methodology itself is subject to scrutiny [3][6].

Future Work
  • Real-Time Monitoring: Extend to real-time mode using the EventLogWatcher API to trigger detection algorithms as events are generated, reducing time-to-detection for active attacks.
  • Multi-System Correlation: Scale the correlation engine across distributed log sources feeding into a centralized database for enterprise lateral movement detection [12].
  • ML Model Improvement: Explore LSTM-based sequence modeling for per-user authentication pattern learning and Transformer architectures for long-range temporal dependencies [2].
  • Active Directory Deep Analysis: Integrate Kerberos ticket-granting events, Group Policy logs, and domain controller replication events for enhanced APT detection coverage [11].
  • Anti-Forensic Resilience: Detect log clearing events (Event ID 1102) and implement secondary evidence preservation mechanisms to reduce impact of anti-forensic tampering [9].
  • SIEM Export Integration: Build native API connectors for Splunk, Microsoft Sentinel, and ELK stack to improve adoption in enterprise SOC environments [3].
